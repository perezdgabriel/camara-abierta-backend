## Phase 2: Data Collection Stabilization

### 2.1 Bill ingestion flow — Completed

Current flow (already implemented):

1. `OpenDataCamaraClient.get_mensajes_x_anno()` + `get_mociones_x_anno()` → list of bulletins
2. `SenadoAsync.fetch_bills_parallel()` → bill detail from Senado
3. `BillParser.parse_bill()` → normalized payload
4. `sync_bill` task → `upsert_bill()` → DB
5. `index_bill` task → Elasticsearch (deferred — skip for v0.1)
6. Voting sessions dispatched from bill data

Completed validation:

- Live dry-run validated the real Senado incremental flow with `python -m app.cli ingestors bills --since 2026-05-20 --dry-run`
- Focused tests now cover explicit `since`, `IngestorState`-driven incremental sync, full-scan fallback, and DB-unavailable fallback
- Parser coverage now includes a representative Senado XML payload through `SenadoClient._parse_bill_xml()`, `BillParser`, and `VoteParser`

### 2.2 Legislator ingestion flow — Completed

Current flow:

1. `SenadoClient.get_senadores_vigentes()` → senators
2. `OpenDataCamaraClient.get_diputados_periodo_actual()` → deputies
3. `run_ingest_reference_data()` → regions and districts used for geography assignment
4. `LegislatorParser` → normalized payload
5. `sync_legislator` task → `upsert_legislator()` → DB

Completed validation:

- Focused tests now cover both sources through `run_ingest_legislators()` dispatch
- Parser coverage verifies deputy current-party selection and senator circumscription normalization
- District assignment is validated against reference-data payloads from `get_regiones()` and `get_distritos()`
- Live OpenData Camara deputy responses still omit `Distrito` nodes in practice, so geography linkage relies on reference-data ingestion rather than inline deputy payloads
- Live end-to-end dry-run across both sources is still pending

### 2.3 Voting session ingestion flow — Completed

Current flow:

1. `SenadoClient.get_bills_by_date()` → bulletins modified since date
2. `SenadoClient.get_votes_by_bulletin()` → votes per bulletin
3. `VoteParser.parse_senate_vote()` → normalized payload
4. `sync_voting_session` task → `upsert_voting_session()` → DB

Completed validation:

- Focused tests now cover explicit `since`-date incremental sync and `IngestorState`-driven voting sync
- Incremental voting ingestion now deduplicates repeated bulletin numbers before fetching Senado vote details
- Write-layer coverage now verifies matched legislator votes are created or updated and unmatched names are ignored

Phase 2 status: Bill ingestion is live-validated, legislator ingestion has focused multi-source coverage plus reference-data geography verification, and voting ingestion now has focused incremental and legislator-vote reconciliation coverage. Real live end-to-end validation is still pending for legislators and voting sessions.

## Phase 4: AI Summaries for Bills

### 4.1 Create bill summary LLM function — Completed

- Add `generate_bill_summary(full_text: str) -> str` to `app/services/llm.py`
- Prompt: plain-language Spanish, 2-3 paragraphs (what it proposes, who it affects, why it matters)
- Use Gemini first, OpenWebUI as fallback (existing pattern)

### 4.2 Create Celery task — Completed

- Add `generate_bill_ai_summary` task to `app/tasks/bills.py`
- Triggered automatically after `sync_bill` completes
- Reads `bill.full_text`, calls LLM, writes `bill.ai_summary` + `bill.ai_summary_updated_at`

### 4.3 Handle missing full text — Completed

- `index_bill` task already extracts PDF text if `full_text` is None and `full_text_url` exists
- AI summary task extracts missing text from `full_text_url` before summarizing
- If no `full_text` and no `full_text_url`, the task skips cleanly

Phase 4 status: Completed in code.
